# Pre-registered evaluation protocol

This protocol is fixed before any baseline is run. Its registration is the public, timestamped
git commit that introduces this file; the commit precedes the commits that add baseline outputs.
No metric, method, or analysis below is changed after seeing baseline results; any deviation is
recorded in `experiments/deviations.md` with its reason.

## Question

On clinical rule sets with known safety properties, what proportion of provable property
violations does each method detect, and how does detection vary with the interaction depth of
the violation?

## Hypotheses

- H1 (primary). Complete verification detects every seeded provable violation; probabilistic and
  sampling methods detect a smaller proportion.
- H2 (depth). For probabilistic and sampling methods, detection decreases as interaction depth
  increases. Complete verification detection does not vary with depth.
- H3 (false alarm). On items with no seeded fault, complete verification and unit testing raise
  no false alarm; a language-model judge may.

## Benchmark

CIV-Bench v0, generated by `benchmark/generate.py` with fixed integer seeds and committed under
`benchmark/suite/`. It contains 180 items: 120 with a seeded violation at interaction depth 1 to
4 (10 items per depth in each of three domains) and 60 with no seeded fault (holds). Domains are
triage (monotonicity), medication (mutual exclusion), and outreach (temporal). Ground truth is
established by the oracle in `verifier.execute` (exhaustive enumeration for decision rule sets,
breadth-first reachability for transition systems), which uses no SMT solver; every violation
witness is confirmed by concrete replay. Interaction depth is the number of conditions or events
that must combine to produce the violation.

## Methods

Every method receives the identical items and returns a verdict, holds or violated, and for a
violated verdict an optional witness.

1. Complete verification: the Z3 harness in `verifier/`. Returns a proof over the full input
   space or a counterexample. Deterministic.
2. Unit-test suite: `baselines/run_unit_test.py`. Draws N = 1000 uniform random inputs (or random
   event sequences up to the bound) per item with a fixed seed, executes the rule set, and
   returns violated if any sampled point violates the property, else holds. This represents
   current quality-assurance practice.
3. Language-model judge: `baselines/render.py` + the judge workflow (`_judge_blind_workflow.js`) + `baselines/ingest_judge.py`. Presents the rule set and property to a
   frontier model and asks for a verdict and, if violated, a witness. Model and decoding are
   fixed (temperature 0); the exact prompt is in the runner and archived with the outputs.
4. NeMo Guardrails: `baselines/run_nemo_guardrails.py`, configured per vendor guidance with a
   fixed language-model backend. Framing in the runner header.
5. Llama Guard: `baselines/run_llama_guard.py`, latest available version. Framing in the
   runner header.

Method availability in this run is recorded in `experiments/environment.md`. A method that
requires infrastructure not available in the run environment (for example a hosted model weight
behind a gated download, or a GPU) is reported as not run, with the exact command to run it; its
result cells are left empty rather than imputed. No result number is fabricated.

## Metrics

For each method:

- Miss rate: among the 120 violated items, the proportion returned as holds. Detection rate is
  one minus the miss rate. This is the primary endpoint.
- False-alarm rate: among the 60 holds items, the proportion returned as violated.
- Witness validity (judge only): among violated verdicts, the proportion whose returned witness
  replays to a concrete violation via `verifier.replay.confirm_witness`.
- Detection rate by interaction depth: detection among violated items at each depth 1 to 4.
- Latency and cost: wall-clock seconds per item, and for model-based methods the token cost.

All proportions are reported with 95% Wilson score confidence intervals. Detection by depth is
also modeled by logistic regression of detection on depth for each probabilistic method, with
the slope and its 95% confidence interval reported. The metric order and method order are
identical across the results text, tables, and figures: complete verification, unit-test suite,
language-model judge, NeMo Guardrails, Llama Guard.

## Determinism and reproducibility

- The released, canonical benchmark is the version-controlled suite under `benchmark/suite*/`;
  the generators are deterministic (seeds derive from a stable sha256 of the construction
  parameters) and document and reproduce an equivalent suite. All results are computed on the
  committed suite.
- The unit-test seed is 20260610; N = 1000.
- The judge runs at temperature 0 with a pinned model id recorded in the outputs.
- Each run writes raw per-item outputs to `experiments/runs/<method>/` with atomic writes and
  resume-from-partial, so an interrupted paid run does not re-incur cost.

## Cost discipline

Model-based methods run a deterministic pilot of 12 stratified items first; the full run proceeds
only after the pilot outputs are inspected. Partial outputs are saved per item.

## Analysis

`experiments/analyze.py` reads the raw per-item outputs and the answer keys and computes the
metrics above. It does not contain hand-entered result numbers; every figure and table is
produced from the raw outputs.
